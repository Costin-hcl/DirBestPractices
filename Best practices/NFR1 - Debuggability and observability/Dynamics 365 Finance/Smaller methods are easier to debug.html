Smaller methods make debugging easier

One way to troubleshoot issues which works even in Production, is to record a trace, replicate the issue and open the trace in Trace Parser. 

The resulting ETL log contain all entry and exit points for all X++ methods, as well as SQL statements executed during the recording. This enables the reconstruction of the call tree and duration of each method.

In order to make issues easier to resolve,  it helps to have smaller functions. As an example, it the function has 500 lines of code and fails at line 400, the only indication we'll see in the Trace Parser log is a failure somewhere. If, instead, we break down the long function into smaller ones, we'll be able to pinpoint the latest smaller function executed and focus the investigation to the following one.
