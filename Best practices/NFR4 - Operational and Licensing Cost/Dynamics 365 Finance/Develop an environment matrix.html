Developing an environment matrix

Microsoft offers some recommendations around environment planning and includes a typical setup depending on the complexity of the project.

An environment matrix builds upon that and shows not only the D365 F&SCM environments but also any other environments that are part of the solution and need to be integrated. It will allow the project team to answer question like:

  • Where will we be performing user acceptance testing (UAT)?
  • How are we testing data migration?
  • Have we purchased enough environments to cover all project phases?
  • Do we have licences for testing integrations with legacy systems?

The format of this matrix is the following:

  • Each row represents an independent set of environments required for a functional reason (e.g. DEV1, DEV2, TEST, SIT, SAT, UAT, PROD, BAU Hotfix)
  • Each column represents a system involved as part of the solution, as identified in a landscape diagram. These can be Dynamics 365 F&SCM, Power Platform Dataverse (CDS), SQL database, Azure resource groups, portals, legacy applications, etc.
  • Each cell will show the actual identifier of the system which will be used for a functional reason. Examples: the URL of the D365 F&SCM system for UAT, the Dataverse environment for DEV, the Azure resource group for Production.
